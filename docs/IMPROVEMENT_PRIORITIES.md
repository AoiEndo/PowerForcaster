## 最優先（短期間で実装・評価可能）

1) 時系列向け交差検証（ローリング・オリジン）と Persistence ベースライン
   - 目的: 評価の信頼性確保と簡易ベースラインの導入
   - 実装案: `sklearn.model_selection.TimeSeriesSplit` を用いる、あるいは expanding/sliding-window を自前実装
   - テスト/評価: 現行線形回帰と比較し、Persistence（`lag_24` をそのまま予測）との差を報告
   - 推奨指標: MAE, RMSE, MASE
   - 難易度: 低

2) LightGBM の単純ベースライン導入（特徴量は現状＋追加ラグ）
   - 目的: 非線形性を手早く取り込むことで精度改善を見極める
   - 実装案: `lightgbm` + CV の学習／予測スクリプト（特徴量は `build_features()` の出力を利用）
   - テスト/評価: TimeSeriesSplit 上で LightGBM vs LinearRegression vs Persistence を比較
   - 推奨指標: MAE, RMSE, sMAPE
   - 難易度: 低〜中

3) 補完フラグとシンプル補完ポリシー
   - 目的: 補完時のバイアス検出と学習の安定化
   - 実装案: `*_imputed` フラグ列を作成、短期欠損は線形補間／前方埋め、列欠損率閾値で削除
   - テスト/評価: 補完あり/なしでモデル性能差を比較、補完フラグの重み確認
   - 推奨指標: MAE
   - 難易度: 低

## 優先（中期、効果期待度高め）

4) ローリング CV ＋ パージング（embargo）実装
   - 目的: ラグ特徴による情報リークを防ぎ、より現実的な評価を行う
   - 実装案: CV スライス間でラグ・ウィンドウ幅に応じたパージを入れる（自前ユーティリティ）
   - テスト/評価: パージ有無でのスコア差を確認（過学習の兆候）
   - 推奨指標: MAE, MASE
   - 難易度: 中

5) カレンダー系特徴とフーリエ項（サイクリック符号化）
   - 目的: 日次／週次の周期性を明示的に表現
   - 実装案: `hour_sin/hour_cos`, `week_sin/week_cos`, 祝日フラグ、前日同時刻平均
   - テスト/評価: 追加前後での CV スコア改善、Feature importance（LightGBM）で寄与確認
   - 推奨指標: MAE, sMAPE
   - 難易度: 低〜中

6) ラグ・ウィンドウの拡充と統計量（mean/std/min/max）
   - 目的: より長期・多様な履歴情報を取り込む

   - テスト/評価: 特徴量削減（SHAP,重要度）で効果のあるウィンドウを絞る
   - 推奨指標: MAE
   - 難易度: 中（特徴量増で計算負荷増）

   - テスト/評価: CV 上の安定性（標準偏差低下）と平均スコア改善

   - 推奨指標: MAE, RMSE
   - 難易度: 高

## 評価指標の拡充（提案）
- 基本: `MAE`, `RMSE`（既存）
- スケール不変・比較容易: `MASE`（季節性比較のため）
- パーセンテージ系: `sMAPE`（対ゼロ敏感性を低くする）
- 不確実性: `Pinball loss`（分位予測）, `PI coverage`（区間カバレッジ）

## 欠損処理ルール（実務テンプレート）
- 短期欠損（<=24時間）: 線形補間または前方埋め（未来データは使用しない）。補完箇所に `*_imputed` フラグを付与。
- 中長期欠損 (>24時間): モデルベース補完（ARIMA 等）を検討、またはその区間を評価から除外。
- 列削除閾値: 欠損率 > 30% の列は評価し、必要なら削除。閾値はプロジェクト毎に調整。

## テスト／再現性／受入基準（推奨ワークフロー）
1. 最初に Persistence baseline + TimeSeriesSplit を実装し、主要データセットでスコアを確立する（受入: baseline を下回る改善があること）。
2. LightGBM を追加し、同じ CV 設定でモデル比較を行う（受入: MAE が統計的に改善、または業務閾値を満たす）。
3. 補完ポリシー導入: 補完フラグがモデル寄与を持つか確認。補完により悪化する場合はポリシーを戻す。
4. 追加特徴量は一括で入れず、Feature importance / SHAP で寄与が低いものを削除していく。
